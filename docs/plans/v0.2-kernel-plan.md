# v0.2 Implementation Plan: Custom Hardened Kernel

> Research and planning document for Lorica Linux v0.2.
> Produced 2026-04-02. This is the authoritative plan for the kernel build.

## Context

Lorica Linux v0.1 ships OS-level hardening (sysctl, GRUB boot params, module blacklist, audit rules) on Debian's stock kernel. v0.2 adds the project's long-term differentiator: a custom hardened kernel built from kernel.org LTS, packaged as an opt-in `lorica-kernel-cloud` .deb. This closes the gaps identified in THREAT_MODEL.md: no signed modules, no struct randomization, no stack leak prevention, stock kernel attack surface.

---

## Research Findings

### Kernel LTS Version Selection

**Finding:** Linux 6.12 LTS is the right target.

- As of April 2026, kernel.org shows 6.12 at version 6.12.80+ with EOL extended to December 2028
- Aligns with Debian 13 (trixie), which ships 6.12 as its default kernel
- linux-hardened patchset actively maintained for 6.12 (latest: v6.12.79-hardened1)
- grsecurity skipped 6.12, supporting 5.15, 6.6, and 6.18 instead
- Kicksecure uses Debian's distribution kernel (6.12 on trixie) by default
- The newer 6.18 LTS (also EOL Dec 2028) is available but would diverge from Debian's kernel

**Decision:** Track one LTS (6.12). Add 6.18 later when Debian 14 approaches. Maintaining two doubles kernel maintenance for marginal benefit.

### Kicksecure Hardened Kernel Analysis

**Finding:** Kicksecure's hardened-kernel repo is stuck on Linux 4.19.122 -- significantly outdated.

Useful as a reference for *which* hardening options to enable, but the config cannot be used directly. Key observations:

- They enable all GCC plugins (RANDSTRUCT, STRUCTLEAK, LATENT_ENTROPY)
- Missing every hardening feature added since 2019 (STACKLEAK improvements, INIT_ON_ALLOC, CFI, lockdown LSM, KFENCE, etc.)
- Build model compiles on user's machine at install time for unique symbols -- impractical for cloud servers
- Uses GCC exclusively, explicitly chose not to switch to Clang
- Module signing is permissive (enabled but not enforced) to avoid breaking their own modules

**For Lorica:** Reference their option choices (updated to modern equivalents). Do not follow their build model, kernel version, or permissive module signing.

### KSPP Config Audit

**Finding:** The project's planned config covers all core KSPP recommendations but is missing ~18 high-priority options.

All 27 originally planned settings are present in KSPP recommendations, with two caveats:
- `GCC_PLUGIN_RANDSTRUCT` has a newer compiler-agnostic successor: `RANDSTRUCT_FULL`
- KSPP recommends `CONFIDENTIALITY` lockdown rather than `INTEGRITY` (we keep INTEGRITY for BASE, CONFIDENTIALITY for HARDENED)

**Missing high-priority options:** SECCOMP/SECCOMP_FILTER, STRICT_DEVMEM/IO_STRICT_DEVMEM, VMAP_STACK, IOMMU_SUPPORT/IOMMU_DEFAULT_DMA_STRICT, DEBUG_WX, RANDOMIZE_KSTACK_OFFSET_DEFAULT, PAGE_TABLE_CHECK/PAGE_TABLE_CHECK_ENFORCED, MODULE_SIG_SHA512, ZERO_CALL_USED_REGS.

**Missing medium-priority options:** UBSAN_BOUNDS+UBSAN_TRAP, SLAB_BUCKETS, RANDOM_KMALLOC_CACHES, LIST_HARDENED, BUG_ON_DATA_CORRUPTION, WERROR.

**Features to explicitly disable:** KEXEC, HIBERNATION, PROC_KCORE, LEGACY_TIOCSTI, COMPAT_BRK, DEVMEM, ACPI_CUSTOM_METHOD, LDISC_AUTOLOAD, USERFAULTFD, BINFMT_MISC, IP_DCCP, IP_SCTP.

Full gap analysis in the "KSPP Gap Analysis" section below.

### GCC Plugins vs Clang CFI

**Finding:** GCC for v0.2, evaluate Clang for v0.3.

KSPP recommendations are architecture-dependent:
- **x86_64:** Recommends GCC + plugins + IBT (hardware Indirect Branch Tracking)
- **arm64:** Explicitly recommends Clang + kCFI + Shadow Call Stack

Reference points:
- Android and ChromeOS: Clang exclusively with kCFI
- Kicksecure: GCC
- grsecurity: Proprietary implementation
- `RANDSTRUCT_FULL` is compiler-agnostic since kernel 6.1
- `CONFIG_GCC_PLUGIN_STACKLEAK` is still GCC-specific (compiler-agnostic replacement: `KSTACK_ERASE`, but may not be in 6.12)

**Decision:** Phase 1 (v0.2) use GCC with all plugins -- matches Debian's toolchain, zero complexity. Phase 2 (v0.3) evaluate Clang for arm64, since Shadow Call Stack and kCFI are compelling on that architecture.

### Module Signing Practical Impact

**Finding:** MODULE_SIG_FORCE does NOT break standard cloud server operations.

- All cloud provider agents (SSM, CloudWatch, cloud-init) are pure userspace
- All cloud hardware drivers (ENA, NVMe, Virtio) are mainlined since Linux 4.9+
- All Docker/K8s modules (overlay, br_netfilter, veth, bridge) are standard in-tree modules signed automatically
- Things that actually break: NVIDIA proprietary drivers, VirtualBox, ZFS-DKMS -- none typical for cloud servers

**Decision:** BASE profile enables module signing in permissive mode (log but don't block). HARDENED profile enforces via `module.sig_enforce=1` boot parameter + `kernel.modules_disabled=1` post-boot.

### Driver Stripping Strategy

**Finding:** Clear include/exclude categories for cloud-only kernel.

**Must include:** Virtio (full suite), AWS ENA + Xen (legacy instances), GCP GVE, Azure Hyper-V, NVMe, SCSI core, device-mapper, ext4, XFS, Btrfs (module), overlayfs, FUSE, netfilter/nftables, IPVS, bridge/veth/macvlan/vxlan, WireGuard, cgroups v2, all namespaces, seccomp-BPF, eBPF (hardened via sysctl), AppArmor, SELinux, audit.

**Must exclude:** Bluetooth, WiFi, NFC, infrared, sound, GPU/DRM, USB HID, webcam, joystick, FireWire, Thunderbolt, all physical NIC drivers, legacy protocols (IPX, Appletalk, DECnet), legacy filesystems (NTFS, HFS, JFS, ReiserFS).

**Key decision:** Keep eBPF enabled (unlike Kicksecure which disables it). Cilium, Falco, and systemd require it. Harden via existing sysctl settings.

### Build Infrastructure

**Finding:** GitHub Actions supports native arm64 builds.

- `ubuntu-24.04-arm` runners are GA since September 2024, free for public repos
- Build both architectures natively -- no cross-compilation needed
- Build time estimate: 30-40 minutes per architecture on 4-core runner (GCC plugins add overhead)
- `make bindeb-pkg` produces standard `linux-image-*` and `linux-headers-*` packages
- Boot testing: virtme-ng + QEMU direct boot takes <30 seconds with KVM

**Config drift detection:** Run `make olddefconfig` in CI and diff against pinned config. New 6.12.x point releases introduce new config options with defaults we may not want. CI step fails/warns if olddefconfig changes anything.

### LKRG Evaluation

**Finding:** Ship as optional package.

- LKRG reached 1.0.0 (September 2025) -- production-ready
- GPLv2 loadable kernel module providing runtime kernel integrity checking
- ~2-2.5% performance overhead
- Kicksecure ships it as `lkrg-dkms`; CIQ Rocky Linux Hardened ships it enabled by default
- Compatible with MODULE_SIG_FORCE and hardened kernel configs
- Only incompatibility: `CONFIG_MODULES=n` (which Lorica does not use)

**Decision:** Ship as pre-built `lorica-lkrg` package, `Suggests:` from `lorica-cloud-server`. Each kernel point release requires a matching LKRG rebuild.

| SLAB_FREELIST_HARDENED | ~0% | Include unconditionally |

## Decisions

### 1. Kernel Version: 6.12 LTS

Target Linux 6.12 LTS (currently at 6.12.80+, EOL December 2028). Track ONE version. Revisit when Debian 14 approaches.

### 2. Compiler: GCC with Plugins

Build with GCC + all security plugins for v0.2. Evaluate Clang for v0.3. Use `RANDSTRUCT_FULL` (compiler-agnostic) instead of `GCC_PLUGIN_RANDSTRUCT`. Use `CONFIG_GCC_PLUGIN_STACKLEAK=y` (still GCC-specific).

### 3. Module Signing: Permissive BASE, Enforced HARDENED

- **BASE:** `MODULE_SIG=y` + `MODULE_SIG_ALL=y` + `MODULE_SIG_SHA512` (log unsigned, don't block)
- **HARDENED:** `module.sig_enforce=1` boot param + `kernel.modules_disabled=1` post-boot

### 4. Driver Stripping

Start from Debian 13's `cloud-amd64`/`cloud-arm64` config and strip down. Include all cloud hypervisor drivers, container networking, and security subsystems. Exclude all desktop hardware, physical NICs, legacy protocols/filesystems.

### 5. LKRG: Optional Package

Ship as `lorica-lkrg` (pre-built module, `Suggests:` from `lorica-cloud-server`). Coupled to kernel releases.

### 6. IMA/EVM: Skip for v0.2

Policy configuration burden too high for cloud workloads. Revisit for v0.3.

### 7. UBSAN_BOUNDS + UBSAN_TRAP: HARDENED Only

Killing processes on undefined behavior is too aggressive for BASE.

## KSPP Gap Analysis

### High Priority (must include)

| Option | Purpose |
|--------|---------|
| `CONFIG_SECCOMP=y` + `CONFIG_SECCOMP_FILTER=y` | Container/K8s requirement, Docker security profiles |
| `CONFIG_STRICT_DEVMEM=y` + `CONFIG_IO_STRICT_DEVMEM=y` | Restrict /dev/mem and /dev/port access |
| `CONFIG_VMAP_STACK=y` | Guard pages for kernel stacks (overflow detection) |
| `CONFIG_IOMMU_SUPPORT=y` + `CONFIG_IOMMU_DEFAULT_DMA_STRICT=y` | DMA attack prevention |
| `CONFIG_DEBUG_WX=y` | Detect writable+executable pages at boot |
| `CONFIG_RANDOMIZE_KSTACK_OFFSET_DEFAULT=y` | Compile-time default for randomize_kstack_offset |
| `CONFIG_PAGE_TABLE_CHECK=y` + `CONFIG_PAGE_TABLE_CHECK_ENFORCED=y` | Detect page table corruption |
| `CONFIG_MODULE_SIG_SHA512=y` | SHA-512 for module signatures |
| `CONFIG_ZERO_CALL_USED_REGS=y` | Clear registers on function return (ROP mitigation) |
| `CONFIG_CFI_CLANG=y` | Defer to v0.3 (Clang required) |

### Medium Priority (should include)

| Option | Purpose | Profile |
|--------|---------|---------|
| `CONFIG_UBSAN_BOUNDS=y` + `CONFIG_UBSAN_TRAP=y` | Runtime bounds checking | HARDENED-ONLY |
| `CONFIG_SLAB_BUCKETS=y` | Isolate slab allocations by size | BASE |
| `CONFIG_RANDOM_KMALLOC_CACHES=y` | Randomize kmalloc cache selection | BASE |
| `CONFIG_LIST_HARDENED=y` | Hardened linked list operations | BASE |
| `CONFIG_BUG_ON_DATA_CORRUPTION=y` | Panic on detected data corruption | BASE |
| `CONFIG_WERROR=y` | Treat compiler warnings as errors | BASE (test first) |

### Features to Explicitly Disable

| Option | Reason |
|--------|--------|
| `CONFIG_KEXEC=n` | Already disabled via sysctl; enforce at compile time |
| `CONFIG_HIBERNATION=n` | No hibernation on servers |
| `CONFIG_PROC_KCORE=n` | Don't expose kernel memory via /proc |
| `CONFIG_LEGACY_TIOCSTI=n` | Disable legacy TTY injection |
| `CONFIG_COMPAT_BRK=n` | Disable heap ASLR bypass |
| `CONFIG_DEVMEM=n` | No /dev/mem on servers |
| `CONFIG_ACPI_CUSTOM_METHOD=n` | Arbitrary ACPI write vector |
| `CONFIG_LDISC_AUTOLOAD=n` | Already disabled via sysctl; compile-time enforcement |
| `CONFIG_USERFAULTFD=n` | Already disabled via sysctl; compile-time enforcement |
| `CONFIG_BINFMT_MISC=n` | Arbitrary format handler registration |
| `CONFIG_IP_DCCP=n`, `CONFIG_IP_SCTP=n` | Already blacklisted via modprobe; compile-time removal |

## Implementation Steps (Dependency Order)

### Phase 1: Kernel Config (Week 1-2)

**Files to create:**
- `packages/lorica-kernel-cloud/kernel-config-x86_64`
- `packages/lorica-kernel-cloud/kernel-config-arm64`
- `packages/lorica-kernel-cloud/debian/`

**Process:**
1. Start from Debian 13's `cloud-amd64` kernel config (not defconfig)
2. Strip excluded driver categories
3. Apply all KSPP options from gap analysis
4. Enable GCC plugins: STACKLEAK, LATENT_ENTROPY
5. Enable RANDSTRUCT_FULL
6. Set module signing: MODULE_SIG=y, MODULE_SIG_ALL=y, MODULE_SIG_SHA512
7. Set lockdown: LOCK_DOWN_KERNEL_FORCE_INTEGRITY
8. Explicitly disable: KEXEC, HIBERNATION, PROC_KCORE, LEGACY_TIOCSTI, COMPAT_BRK, BINFMT_MISC, etc.
9. Test-build to verify config compiles cleanly
10. Repeat for arm64 starting from Debian 13's `cloud-arm64` config

### Phase 2: Build Infrastructure (Week 2-3)

**File:** `.github/workflows/build-kernel.yml`

**CI pipeline:**
1. Download 6.12.x tarball from kernel.org (verify PGP signature)
2. Build x86_64 on `ubuntu-latest` (native)
3. Build arm64 on `ubuntu-24.04-arm` (native, free for public repos)
4. `make bindeb-pkg LOCALVERSION=-lorica KDEB_PKGVERSION=6.12.x-lorica-1`
5. Config drift detection: `make olddefconfig` + diff (fail if config changes)
6. ccache via `actions/cache` across runs
7. ~30-40 minutes per architecture

**Kernel source management:**
- Download tarball per build (not a git submodule)
- Pin version in `KERNEL_VERSION` file at repo root
- GitHub Action cron watches kernel.org for new 6.12.x releases

### Phase 3: Boot Testing (Week 3-4)

**Boot test in CI:**
- virtme-ng + QEMU direct boot (<30s with KVM)
- Verify: boots, reaches userspace, sysctls applied, lockdown active

**VM smoke test:**
- Extend `tests/vm-smoke-test.sh` with kernel-specific phases
- Install .deb, reboot, verify Docker/containers/cloud agents
- Verify `uname -r` shows `6.12.x-lorica`

### Phase 4: Debian Packaging (Week 4)

**Package:** `lorica-kernel-cloud`
- Architecture: `amd64`, `arm64`
- Coexists with Debian's kernel (no Replaces/Conflicts)
- Postinst: `update-grub`
- Add `Suggests: lorica-kernel-cloud` to `lorica-cloud-server`

### Phase 5: LKRG Package (Week 5)

**Package:** `lorica-lkrg`
- Pre-built module (not DKMS), compiled against each kernel release
- Auto-loaded via systemd service or modprobe.d
- `Suggests:` from `lorica-cloud-server`
- Each kernel point release requires matching LKRG rebuild

### Phase 6: Documentation (Week 5-6)

All docs updated before v0.2 tag:

| File | Changes |
|------|---------|
| `docs/THREAT_MODEL.md` | Remove v0.1 limitations section, add v0.2 kernel hardening section, add LKRG |

### Phase 7: Testing (Week 6)

- Boot: x86_64 and arm64 (QEMU CI + real VM)
- Docker/K8s: containers, networking, overlay module
- Cloud agents: SSM, cloud-init
- Compatibility: nginx, PostgreSQL, Node.js on Lorica kernel
- Performance: boot time within 15% of stock Debian
- Module signing: log/block behavior per profile
- LKRG: loads, integrity checks, no false positives

## Risk Assessment

### High Risk

1. **Kernel config iteration.** Missing a single driver means the VM won't boot. **Mitigation:** Start from Debian 13's cloud config and strip down, not build up from defconfig.

2. **GCC plugin build failures.** RANDSTRUCT and STACKLEAK can cause obscure build errors. **Mitigation:** Build early, build often. Fix or disable problematic drivers.

3. **CI build time.** 30-40 minutes per architecture, ~80 minutes total. **Mitigation:** ccache via `actions/cache`; strip aggressively; consider larger runners if builds creep past 45 minutes.

### Medium Risk

4. **DKMS/third-party module breakage.** RANDSTRUCT_FULL breaks out-of-tree module ABI. **Mitigation:** Ship linux-headers; document clearly; LKRG build is controlled.

5. **arm64 config differences.** Different defaults and requirements. **Mitigation:** Separate configs per architecture; test both in CI.

### Low Risk

6. **Module signing key management.** **Mitigation:** Ephemeral key per build (standard practice).

## Resolved Questions

1. **AWS Xen:** Include. Binary size cost is trivial. Strip in v0.3 if unused.
2. **Btrfs:** Module (not built-in).
3. **IMA/EVM:** Skip for v0.2. Policy burden too high for cloud ROI.
4. **UBSAN_BOUNDS + UBSAN_TRAP:** HARDENED only.

2. **WERROR:** May cause build failures with GCC warnings in third-party drivers.
3. **Virtio serial console:** Verify it works after stripping physical serial drivers.
